## <span style="color: #2C3E50;"><span style="color: #2C3E50;">部署流程的健壮性与错误处理：确保每一次发布都稳如磐石</span></span>



在我长期维护 GitHub Pages 博客的实践中，我发现，一个“一键发布”的脚本，仅仅是解决了“便捷性”的问题，更关键的在于其“可靠性”。设想一下，你在深夜奋笔疾书，好不容易写完了一篇技术干货，满心欢喜地运行了你的发布脚本，结果因为网络波动，或者 Git 仓库暂时不可用，导致发布失败，甚至更糟，部分文件被错误地推送。这种体验无疑是毁灭性的。因此，在我迭代脚本的过程中，投入了相当大的精力来构建健壮的错误处理机制。

首先，对于 Git 操作，我并没有简单地依赖 `subprocess.run()` 的返回值。我会捕获 `CalledProcessError` 异常，并在捕获到异常后，详细分析错误信息。例如，如果 `git push` 失败，我会检查是由于网络问题、权限问题，还是远程仓库冲突。根据不同的错误类型，脚本会给出更有针对性的提示，而不是仅仅显示一个通用的“发布失败”。我还会实现一个简单的“回滚”机制，当某个关键步骤（如 `git push`）失败时，脚本可以尝试撤销已经执行的部分操作，避免造成数据不一致。*这确保了即使在部署过程中出现意外，你的博客仓库也能尽可能地保持在一个干净、可用的状态。*

另外，对于静态站点生成器的构建过程（例如 `jekyll build`），我也加入了超时机制和日志记录。如果生成过程卡死，脚本会及时中断并报错，防止无限期等待。生成的静态文件通常会输出到 `_site` 这样的目录。在脚本中，我会先清理旧的 `_site` 目录，再重新生成，避免因为增量更新带来的潜在问题。之后，我会将 `_site` 目录下的内容添加到 Git 暂存区，并且 **只添加 `_site` 目录下的内容**。这一点非常重要，因为它避免了将项目本身的源码（如 Markdown 文件、配置文件等）错误地推送到 `main` 分支，确保了部署分支的纯净。

*   **错误捕获与日志记录：** 详细记录每次部署的操作，特别是 Git 命令的输出，以便快速定位问题。
*   **网络状态检测：** 在执行 Git 操作前，可以尝试 ping 一下 GitHub 的服务器，或者检查网络连接，避免无谓的尝试。
*   **幂等性设计：** 确保脚本的多次运行不会产生副作用，例如重复添加文件或生成重复内容。
